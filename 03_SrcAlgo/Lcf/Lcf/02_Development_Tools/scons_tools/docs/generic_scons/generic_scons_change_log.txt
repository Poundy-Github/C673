                    - keep the file updated
                    - wrap lines after 100 characters
                    - add latest entries at the top (reverse order by date)
                    - use the format described below for all entries
                    - write only change descriptions that have an impact to the user and avoid 
                      technical details.
                    - REFER TO MKS TICKET ID in the description whenever possible!! Use the format
                      "Type #nnnnnn", e.g. "Issue #123456".
                  
                  Entry Format:
                  -------------
                  YYYY-MM-DD/full name (conti-ID):        => Date format is ISO 8601.
                                                          => Name = conti e-mail address without
                                                                    "@continental-corporation.com"
                  <2_spc>[action]                         => action = {DEL | ADD | FIX | MOD}
                                                             DEL -> feature has been removed
                                                             ADD -> new features created
                                                             FIX -> an issue has been fixed
                                                             MOD -> all other modifications
                  <4_spc> + Description 1
                          + Description 2 ....
                            ... goes on here
                          + Decription 3
                          
                  For each entry multiple types of changes may be combined. Example:
                  
                  2014-02-25/Alex.Foo (cont1234):
                    [FIX]
                      + Fixed script crash when compiled for target CortexA15 (Issue #123456).
                    [ADD]
                      + Added console messages printing detailed information about CortexA15 build 
                        process.
